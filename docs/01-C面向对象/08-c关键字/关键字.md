# 关键字

## 1.volatile

​	作用为让被volatile修饰的内容禁止被编译器优化，它告诉编译器被修饰的内容随时都会被意外更改，比如寄存器更新，中断服务程序更改，使用volatile关键字后每次使用被修饰的内容时都会去对应的内存地址中读取，而不是使用寄存器中缓存的内容。

* volatile解决的是编译器不把修饰的内容放入寄存器中的优化，而不是cpu的缓存优化。
* volatile不是线程安全的，多个线程使用时不保证原子性。

### 1.1 典型使用场景

#### 1.1.1 内存映射（MMIO)

​	嵌入式开发中最常见的场景。硬件寄存器被映射到固定内存地址，外设随时可能改变其值。

```c
/* 假设 0x4000_0000 是串口状态寄存器 */
#define UART_STATUS (*(volatile uint32_t *)0x40000000)

/* 等待发送缓冲区为空 */
while ((UART_STATUS & TX_EMPTY_BIT) == 0) {
    /* 编译器不会把 UART_STATUS 优化到寄存器 */
	/* 每次循环都真实读取硬件寄存器 */
}
```

* 若没有使用volatile对状态寄存器进行修饰的话，编译器会把状态寄存器的值作为固定值一直使用固定值进行while循环，当硬件改变状态寄存器的值时读取到的值还是没有改变的值，所以需要使用volatile对硬件寄存器映射到固定内存地址使每次使用时都到固定内存地址获取。

#### 1.1.2 中断服务程序（ISR）修改的变量

​	主程序与中断服务程序共同使用到的变量可能会随时被中断服务程序意外修改，所以需要使用到volatile进行修饰。

```c
volatile uint32_t g_irq_count = 0;  ///< 中断计数

/* 中断服务程序 */
void TIM2_IRQHandler(void) {
    g_irq_count++;  ///< 中断中修改
}

/* 主循环 */
int main(void) {
    uint32_t last_count = 0;
    while (1) {
        if (g_irq_count != last_count) {  ///< 必须去内存读最新值
            last_count = g_irq_count;
            printf("IRQ triggered! Total: %lu\n", last_count);
        }
    }
}
```

#### 1.1.3 多线程“只读/只写“标志位

​	在多线程的环境中，使用volatile可以防止编译器缓存导致线程看不到标志位的更新。

```c
volatile int g_data_ready = 0;
int g_shared_data;

/* 线程A：生产者 */
void producer(void) {
    g_shared_data = calculate();   ///< 先写数据
    g_data_ready = 1;              ///< 再设置标志
}

// 线程B：消费者
void consumer(void) {
    while (!g_data_ready) {        ///< 轮询标志
        ///< 忙等待
    }
    use(g_shared_data);
}
```

* 虽然在多线程的环境下可以使用，但是不推荐进行使用，因为volatile不保证线程安全，它不保证原子性与跨核心内存序保证。
* 在多线程同时读取或者写入时会产生资源竞争，不保证原子性。
* volatile只是对编译器进行约束，但是在多核环境下一个核对修饰的内容进行更改后，另一个核进行读取时有可能也要从自己的缓存中读取到旧数据，而不是更改后的值。

#### 1.1.4 信号处理函数

​	类似于中断服务程序，信号处理函数本质上是一个**异步回调**。它可以在主程序执行的**任意指令之间**被插入执行。所以也需要使用volatile进行修饰。

```c
volatile sig_atomic_t g_sig_received = 0;

void handler(int sig) {
    g_sig_received = 1; ///< 异步安全写入
}

int main() {
    signal(SIGINT, handler);
    while (!g_sig_received) { ///< 主循环轮询，不会被编译器优化掉
        pause();
    }
}
```





## 2.const

​	常量修饰符const也是被经常使用的关键字之一，它的作用为被const修饰的内容在编译后只能进行读取，不能进行修改，如果试图进行修改的话会触发编译错误。

### 2.1 修饰普通变量

* 使用const对全局变量进行修饰后，这个全局变量就只有可读性了，储存变量的位置从栈中被移动到常量区中，此时对这个常量变量的值进行修改就会触发编译错误。当修饰局部变量是储存位置依然是在栈上，只是只有可读性了。
* 在进行常量定义时必须要进行初始化，如果不对其进行初始化的话就会触发编译错误，因为后续不能对其值进行修改了。



### 2.2 修饰指针

​	如果使用const对指针进行修饰的话有两种修饰方式，一种修改过后指针变为指针常量，一种进行修饰后变为常量指针，这两中修饰的作用是不同的，const在`*`左边的是指针常量，const在`*`的右边的是常量指针。

* **指向常量的指针（const int *p)：**此时的指针p是一个指针常量，它修饰的是指针指向的内容，此时指向的内容不可变，但是指针可以更改。
* **常量指针（int * const p)：**此时p为一个常量指针，修饰的是指针，此时指向的内容可以更改，但是指针不能更改。
* **如果为（const int *const p)：**此时p指向的内容不可变，指针也不可变



### 2.3 修饰函数参数

​	使用const修饰函数的参数是告诉调用者不会修改调用者传入的参数，同时让编译器进行检查，如果进行修改的话编译器就会触发编译错误。



## 3.static

​	static关键字作用为使被修饰的变量或者函数的生命周期与作用域（链接属性）发生改变，它有两种核心使用场景。

### 3.1 修饰局部变量

* 当使用static修饰局部变量时，局部变量的什么周期从函数调用开始到函数调用结束变为程序运行结束，作用域不变，依然只能在函数内部进行使用。储存的位置从栈上变为静态数据区进行储存。
* static修饰的局部变量在函数中只进行一次初始化，如果没有进行显示初始化的话编译器默认赋值为0。



### 3.2 修饰全局变量/函数

* 当使用static对全局变量/函数进行修饰后，全局变量/函数的生命周期不变，但是作用域发生改变，变为仅内部可见（仅进行内部链接），外部不可见（不进行外部链接），此时可以防止全局变量/函数重名，跨文件调用。



## 4.extern

​	extern关键字是跨文件数据共享与解决声明迅速的关键工具，使用它进行修饰的变量只进行声明不进行定义，即不进行内存分配，它只告诉编译器有这样一个东西，在链接时进行绑定就行。它有两个核心应用场景。

### 4.1 跨文件数据共享

​	在 `a.c` 中定义 `int score = 100;`，在 `b.c` 中如果想用，必须写 `extern int score;`（通常放在头文件 `a.h` 中声明）。

### 4.2 解决声明顺序

​	如果 `main` 函数在文件顶部，而全局变量定义在底部，可以在 `main` 前写 `extern int g_val;` 来提前声明使用。
