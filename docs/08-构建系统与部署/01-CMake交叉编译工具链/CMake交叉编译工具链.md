## 1.交叉编译核心概念与底层原理

### 1.1 交叉编译本质

​	交叉编译的本质为在一个平台生成另外一个平台运可执行代码的编译过程。



### 1.2 交叉编译的关键要素

* **构建平台（build）：**编译器实际运行的机器。

* **运行平台（HOST）：**交叉编译后的可执行代码运行的机器。
* **目标平台（target）：**编译的生成的代码所针对架构（ARMCortex_M、RISC-V等架构）。



### 1.3 交叉编译存在的原因

* **目标平台无法进行编译：**如裸机MCU没有操作系统，无法运行编译器。
* **目标平台性能不足：**如手机、路由器等设备编译大型项目太慢。
* **目标平台环境缺失：**如缺乏足够的储存、内存或者开发工具链。
* **多平台统一构建：**可以在一个构建平台上一次性编译出多个不同平台架构版本的可执行代码。



### 1.4 与平台编译的区别

| 普通编译               | 交叉编译                                         |
| ---------------------- | ------------------------------------------------ |
| 编译器和目标代码同架构 | 编译器和目标代码异架构                           |
| 可直接运行编译结果     | 编译结果需部署到目标机运行                       |
| 系统头文件/库直接可用  | 需要目标平台的 sysroot（头文件、库、链接器脚本） |
| 链接时使用本机库       | 链接时需指定目标平台的库路径                     |



## 2.交叉编译的核心组件

### 2.1 编译器

* **作用：**将高级语言翻译为目标平台的汇编语言。

* **核心工作流程：**现代编译器编译的过程分为三个阶段，对应编译前端、编译中端、编译后端。
  * **编译前端：**作用为理解源程序、验证其合法性，并将其转换为与机器无关的中间表示（IR）的模块，需要结果词法分析、语法分析、语义分析、中间件代码生成等处理得到与机器无关的中间件表示的模块。
  * **编译中端：**作用是对中间件表示的模块进行优化，进行常量传播、死代码消除、循环展开等优化，从而让程序跑得更快、体积更小、能耗更低，同时保证语义不变。
  * **编译后端：**将经过优化后的中间件表示映射到具体的硬件平台指令集，生成对应平台的汇编代码或者机器码。



### 2.2 汇编器

* **作用：**将人类可读的汇编语言转化为机器可执行的二进制机器码。
* **工作流程：**汇编器的工作流程相对于编译器来说简单的多，通常分为两遍扫描。
  * **第一遍扫描：**作用为构建符号表，通过扫描所有标号（Label）和符号定义、记录每个符号对应的内存地址或偏移量，来构建符号表
  * **第二遍扫描：**作用为二进制代码的生成与重定位，根据符号表将对应的助记符与符合引用替换为对应的操作码与具体的数值地址，生成对应的可执行二进制机器码文件。



### 2.3 链接器

* **作用：**将一个或多个由汇编器/编译器生成的目标文件（.0、obj）和库文件（.a、.lib、.so），合并为一个完整的、可被加载执行的可执行文件或者共享库。
* **链接器的核心任务：**
  * **符合解析：**
    * **问题：**每个 `.o` 文件可能引用了在其他文件中定义的函数或变量。
    * **解决方法：** 链接器扫描所有输入文件，建立全局符号表，将每个**未定义符号**与某个**定义符号**精确匹配。
  * **重定位：**
    * **问题：**编译器/汇编器生成目标文件时，不知道代码最终在内存中的确切地址，因此使用占位符（如 0x0）表示外部引用地址。
    * **解决方法：** 链接器确定每个段（Section）的最终内存布局后，**回填**所有占位符为真实地址。
    * **类型：**
      * **静态重定位：** 链接时一次性完成（传统方式）
      * **动态重定位：** 运行时由动态链接器完成（用于共享库）。
  * **段合并与内存布局：**
    * **作用：**将所有 `.o` 文件中同名的段（如 `.text`, `.data`, `.bss`）合并。
    * 按照目标平台的格式规范（ELF / PE / Mach-O）组织最终的内存映像结构。
    * 处理对齐要求，插入填充字节。



### 2.4 C库

* **作用：**增加代码的可移植性，只要遵循 C 标准，同一份代码可在任何有合规 libc 的平台上编译运行。平衡安全与性能，经过数十年优化的 `memcpy`、`sort` 等函数通常比手写版本更快更安全。抽象不同系统，屏蔽了不同 OS 的系统调用差异，让开发者专注于业务逻辑。所有高级语言的基石， Python、Ruby、Go、Rust 等的运行时都直接或间接依赖 C 库。
* **C库的核心组成：**

| 头文件       | 核心功能         | 典型函数/宏                              |
| ------------ | ---------------- | ---------------------------------------- |
| `<stdio.h>`  | 输入输出         | `printf`, `scanf`, `fopen`, `fclose`     |
| `<stdlib.h>` | 通用工具         | `malloc`, `free`, `atoi`, `exit`, `rand` |
| `<string.h>` | 字符串/内存操作  | `strlen`, `strcpy`, `memcpy`, `strcmp`   |
| `<math.h>`   | 数学运算         | `sin`, `cos`, `sqrt`, `pow`              |
| `<time.h>`   | 日期时间         | `time`, `clock`, `difftime`, `strftime`  |
| `<errno.h>`  | 错误处理         | `errno`, `perror`, `strerror`            |
| `<assert.h>` | 调试断言         | `assert`                                 |
| `<stdint.h>` | 精确宽度整数类型 | `int32_t`, `uint64_t`, `SIZE_MAX`        |

* **C库的使用：**
  * **编译阶段：** 编译器通过 `#include` 获取 C 库的声明（函数原型、类型定义），用于类型检查和代码生成。
  * **链接阶段：** 链接器将你的代码中对 `printf`、`malloc` 等的调用，解析到 C 库的实际实现上。
  * **运行阶段：** 若使用动态链接，操作系统在程序启动时自动加载 `libc.so`；若静态链接，则 C 库代码已被复制进可执行文件。

### 2.5 GDB(调试器)

* **作用：**GDB是 GNU 工具链中的交互式源码级调试器, GDB 负责在程序运行时提供透视能力，使得能够观察、控制并诊断程序的内部状态。
* **GDB的核心功能：**
  * **断点控制：** 在指定代码行、函数入口或特定条件满足时暂停执行。
  * **单步执行：** 逐行（`next`）或逐指令（`stepi`）运行，深入函数内部或跳过函数调用。
  * **状态检视：** 实时查看变量值、寄存器内容、内存布局、调用栈帧。
  * **动态修改：** 在暂停时修改变量值、寄存器甚至跳转执行流，用于验证修复假设。
* **GDB 的工作原理：**
  * **ptrace 系统调用：**GDB 通过 Linux 的 `ptrace` 系统调用附加到目标进程，从而获得：
    * 读取/写入目标进程的内存和寄存器。
    * 拦截信号（如 `SIGTRAP` 实现断点）。
    * 控制进程的执行（暂停、继续、单步）。
  * **调试信息：**编译器在生成目标文件时，可通过 `-g` 选项嵌入调试信息格式的调试元数据，包括：
    * 源码行号 ↔ 机器指令地址的映射。
    * 变量名 ↔ 内存偏移/寄存器的映射。
    * 类型定义、作用域、内联展开信息等。
  * **符号表：**即使不启用完整调试信息，未剥离的二进制仍保留函数名和全局变量符号，使 GDB 至少能识别函数边界。



### 2.6 构建系统

* **作用：**构建系统能够自动化管理代码编译、链接及依赖关系的基础设施。在大型项目中我们不可能直接使用gcc -c a.c && gcc -c b.c && gcc a.o b.o -o app等操作来进行项目构建，而且也没有足够的能力来进行维护，所以就有了构建系统来帮助开发者完成系统构建。
* **构建系统的核心职责：**

| 职责       | 说明                                                   |
| ---------- | ------------------------------------------------------ |
| 依赖图构建 | 解析源文件、头文件、库之间的依赖关系，生成有向无环图   |
| 增量编译   | 基于文件时间戳或内容哈希，仅重建受影响的节点           |
| 并行调度   | 利用多核 CPU 同时编译无依赖关系的文件（如 `make -j8`） |
| 工具链抽象 | 屏蔽不同 OS/编译器/架构的差异，提供统一接口            |
| 产物管理   | 定义输出目标（可执行文件、库、安装包等）及其生成规则   |

* **主流构建系统演进与对比：**

| 构建系统    | 类型        | 特点                                              | 适用场景                      |
| ----------- | ----------- | ------------------------------------------------- | ----------------------------- |
| Make        | 命令式      | 历史悠久，基于文件时间戳，语法简单但易出错        | Unix/Linux 传统项目、小型项目 |
| CMake       | 元构建      | 生成 Make/Ninja/VS 工程，跨平台事实标准，生态最强 | C/C++ 跨平台项目、库开发      |
| Ninja       | 后端执行器  | 极简、极速，专注并行执行，通常由 CMake/Meson 生成 | 大型项目加速构建              |
| Meson       | 声明式      | 现代设计，原生支持 Ninja，语法友好，启动快        | 新项目、GNOME/GStreamer 生态  |
| Bazel       | 声明式+缓存 | Google 出品，支持远程缓存、多语言、超大规模仓库   | 巨型单体仓库、微服务集群      |
| Gradle/Buck | JVM 生态    | Android/Java 主导，支持增量+缓存                  | Android、JVM 项目             |



## 3.源码到可执行文件流水线

### 3.1 预处理

* **输入：**源代码。
* **工具：**预处理器。
* **核心操作：**
  * 展开 `#include`：递归插入头文件内容（包括 C 库头文件如 `<stdio.h>`）。
  * 宏替换：展开 `#define` 定义的宏。
  * 条件编译：处理 `#ifdef` / `#ifndef` / `#endif`。
  * 删除注释、添加行号标记。
* **输出：** 纯净的、不含预处理指令的 C/C++ 源码（`.i`）。

### 3.2 编译

* **输入：** 预处理后的源码。
* **工具：** 编译器。
* **核心操作：**
  - **编译前端：**生成平台无关的中间表示
  - **编译中端：**常量折叠、死代码消除、内联、循环优化等
  - **编译后端：**指令选择、寄存器分配、指令调度、后端优化。
* **输出：**汇编代码（`.s`）

### 3.3 汇编

* **输入：** 汇编代码（`.s`）

* **工具：** 汇编器（`as`, `nasm`）

* **核心操作：**

  - 将每条汇编指令编码为机器码。

  - **生成段（Section）**：`.text`（代码）、`.data`（已初始化数据）、`.bss`（未初始化数据）、`.debug_*`（调试信息）。

  - **生成重定位表**：标记所有外部符号引用和待回填地址的位置。

  - **生成符号表**：导出本文件定义的全局符号，记录未定义的外部符号。

**输出：** 目标文件（`.o` / `.obj`）。



### 3.4 链接

* **输入：** 多个`.o`文件 + 静态库（`.a`）/动态库（`.so` / `.dll`）。

* **工具：** 链接器（`ld`, `lld`, `link.exe`）。

* **核心操作：**

  - **符号解析：** 将所有未定义符号匹配到唯一定义（来自其他 `.o` 或 C 库）。

  - **段合并：** 同名段合并，按内存布局规范排列。

  - **重定位：** 根据最终内存布局，回填所有占位符为真实地址。

  - **生成入口点：** 设置 `_start` / `main` 入口，初始化 C 运行时环境。

* **输出：** 可执行文件（ELF / PE / Mach-O）或共享库。

### 3.5 加载运行

* **输入：** 可执行文件。

* **工具：** OS 加载器 + 动态链接器（`ld-linux.so` / `dyld`）。

* **核心操作：**

  - 创建进程，映射可执行文件到虚拟内存。

  - 若依赖共享库（包括 `libc.so`），动态链接器解析并加载。

  - 执行动态重定位（修复共享库中的地址引用）。

  - 调用 C 运行时初始化代码（初始化堆、全局变量、atexit 链表等）。

  - 跳转至 `main()` 函数。

**输出：** 运行中的进程。



## 4.QUESTION

### 4.1 交叉编译与本地编译的区别？

​	**交叉编译是为了让存在目标平台与构建平台的架构不同、目标平台没有编译的能力、目标平台性能不足、目标平台环境缺失的设备可以运行对应的代码而存在的编译。本地编译只是为了能够在自身平台上能够运行而存在。**

| 维度                     | 本地编译（Native Compilation） | 交叉编译（Cross Compilation）            |
| ------------------------ | ------------------------------ | ---------------------------------------- |
| **构建平台 vs 目标平台** | 相同（Build = Host = Target）  | 不同（Build ≠ Target）                   |
| **编译器运行位置**       | 目标平台自身                   | 性能更强的主机/开发机                    |
| **编译结果运行位置**     | 编译器所在的同一台机器         | 异构的目标设备                           |
| **典型场景**             | PC上编译PC程序                 | PC上编译手机/嵌入式/路由器程序           |
| **工具链**               | 原生编译器（如 `gcc`）         | 交叉编译器（如 `aarch64-linux-gnu-gcc`） |
| **依赖处理**             | 直接使用本机系统库             | 需要目标平台的 sysroot（头文件、库）     |
| **调试方式**             | 本机直接运行调试               | 远程调试（GDB Server）、串口、仿真器     |

### 4.2 不同架构在底层指令集与系统调用的差异？、

​	**不同架构在指令集与系统调用上的差异是本质性的：CISC vs RISC 的设计哲学决定了指令编码、寄存器体系和访存模式的不同；而系统调用作为软硬件交界，其调用约定、陷入指令和调用号表因架构历史与ABI规范而异。编译器后端和操作系统内核的职责，正是将这些差异封装隔离，让上层应用能够以统一的高级接口编程。**

| 特性           | x86/x86-64 (CISC)              | ARM64 / RISC-V (RISC)              |
| -------------- | ------------------------------ | ---------------------------------- |
| **指令长度**   | 变长（1-15字节）               | 定长（4字节，RISC-V可选压缩2字节） |
| **指令复杂度** | 复杂，单条指令可完成多操作     | 简单，单周期执行                   |
| **访存指令**   | 任意指令都可访存               | 只有 Load/Store 指令能访存         |
| **寄存器数量** | 较少（x86-64有16个通用）       | 较多（ARM64有31个，RISC-V有32个）  |
| **编码复杂度** | 高，需解析前缀/操作码/ModR/M等 | 低，字段位置固定                   |

* **寄存器约定差异：**编译器后端必须按目标平台的调用约定生成代码，否则函数调用会崩溃。

| 架构                  | 调用者保存                      | 被调用者保存                | 参数传递                   | 返回值    |
| --------------------- | ------------------------------- | --------------------------- | -------------------------- | --------- |
| **x86-64 (System V)** | rax, rcx, rdx, rsi, rdi, r8-r11 | rbx, rbp, r12-r15           | rdi, rsi, rdx, rcx, r8, r9 | rax, rdx  |
| **x86-64 (Windows)**  | rax, rcx, rdx, r8-r11           | rbx, rbp, rdi, rsi, r12-r15 | rcx, rdx, r8, r9           | rax, xmm0 |
| **ARM64 (AArch64)**   | x0-x18                          | x19-x28                     | x0-x7                      | x0, x1    |
| **RISC-V**            | a0-a7, t0-t6                    | s0-s11                      | a0-a7                      | a0, a1    |

* **系统调用差异：**系统调用是用户程序请求内核服务的唯一入口，不同架构的**调用方式、调用号、参数传递**完全不同。

  * **调用指令：**

  | 架构       | 陷入内核的指令                                  | 说明                          |
  | ---------- | ----------------------------------------------- | ----------------------------- |
  | **x86**    | `int 0x80`（ legacy ） / `sysenter` / `syscall` | `syscall` 是现代64位标准      |
  | **x86-64** | `syscall`                                       | 快速系统调用，使用 MSR 寄存器 |
  | **ARM32**  | `swi #0` / `svc #0`                             | Software Interrupt            |
  | **ARM64**  | `svc #0`                                        | Supervisor Call               |
  | **RISC-V** | `ecall`                                         | Environment Call              |

  * **系统调用号差异：**

  | 系统调用 | x86-64 | ARM64 | RISC-V |
  | -------- | ------ | ----- | ------ |
  | `read`   | 0      | 63    | 63     |
  | `write`  | 1      | 64    | 64     |
  | `exit`   | 60     | 93    | 93     |
  | `open`   | 2      | 56    | 56     |
  | `close`  | 3      | 57    | 57     |
