# 02 — 框架选型与搭建

## 1. 框架对比

### Unity + FFF（本项目采用）

| 维度 | 评价 |
|------|------|
| 集成难度 | ⭐ 极低 — 两个头文件即可 |
| 学习成本 | ⭐⭐ 低 — 宏 API，IDE 可自动补全 |
| Mock 生成 | 手写头文件（1 模块 ≈ 30 行） |
| 编译依赖 | 零 — 宿主机 GCC |
| CI 友好 | 优秀 — `cmake --build && ./test` |

**适合**：中小型项目（< 50 个外设驱动模块），团队熟悉 C 宏。

### Unity + CMock

| 维度 | 评价 |
|------|------|
| 集成难度 | ⭐⭐⭐ 中 — 需要 Ruby 运行环境 |
| Mock 生成 | 自动 — 从 `.h` 文件生成 mock `.c` |
| 灵活性 | 低 — 生成代码不鼓励手动修改 |
| 编译依赖 | Ruby + CMake |

**适合**：大型项目，大量 API 需要 mock，愿意维护 CMock 配置。

### CppUTest

| 维度 | 评价 |
|------|------|
| 语言 | C++，但可测 C 代码 |
| Mock 能力 | 内置 `mock().expectOneCall()` |
| 适用场景 | 混合 C/C++ 项目 |

**适合**：团队习惯 C++，项目本身混用 C/C++。

### 选型建议

```
项目规模 < 10 个驱动模块  →  Unity + FFF
项目规模 10~50 个驱动模块 →  Unity + FFF（仍可管理）
项目规模 > 50 个驱动模块  →  Unity + CMock
C++ 项目                   →  CppUTest 或 GoogleTest
```

## 2. 目录结构

```
project/
├── components/pal/                  # 被测代码
│   └── pal_gpio/
│       ├── include/pal_gpio.h
│       └── pal_gpio.c
└── tests/                            # 测试根目录（与 components 平级）
    ├── CMakeLists.txt                 # 测试工程 CMake
    ├── fff_globals.c                  # FFF 全局定义（有且仅有一个）
    ├── test_pal_xxx.c                 # 测试用例文件
    ├── unity/                         # Unity 框架（3 个文件）
    │   ├── unity.c
    │   ├── unity.h
    │   └── unity_internals.h
    ├── fff/                           # FFF 框架（1 个文件）
    │   └── fff.h
    └── mocks/                         # Mock 头文件（替换 ESP-IDF 头文件）
        ├── esp_err.h                  #   基础类型定义
        ├── esp_log.h                  #   日志宏
        └── driver/                    #   对应 ESP-IDF driver/ 目录
            ├── gpio.h                 #   GPIO Mock
            ├── i2c_master.h           #   I2C Mock
            ├── spi_master.h           #   SPI Mock
            └── ...
```

**原则**：Mock 头文件的目录结构**镜像** ESP-IDF 的 include 层级。当被测代码写 `#include "driver/gpio.h"` 时，CMake 的 `-I` 路径让编译器先找到 `tests/mocks/driver/gpio.h`。

## 3. CMakeLists.txt

```cmake
cmake_minimum_required(VERSION 3.16)
project(pal_test LANGUAGES C)

set(CMAKE_C_STANDARD 17)
set(CMAKE_C_FLAGS "${CMAKE_C_FLAGS} -Wall -Wextra -Werror -g -O0")

# 被测代码需要的芯片能力宏（Mock 环境下填合理默认值）
add_compile_definitions(
    SOC_LEDC_TIMER_NUM=4
    SOC_LEDC_CHANNEL_NUM=8
)

add_executable(pal_test
    unity/unity.c                   # Unity 框架
    fff_globals.c                    # FFF 全局定义
    test_pal_gpio.c                  # 测试用例
    ../components/pal/pal_gpio/pal_gpio.c  # 被测源码
)

target_include_directories(pal_test PRIVATE
    ${CMAKE_SOURCE_DIR}/fff           # fff.h
    ${CMAKE_SOURCE_DIR}/unity         # unity.h
    ${CMAKE_SOURCE_DIR}/mocks         # Mock 头文件（优先于 ESP-IDF）
    ${CMAKE_SOURCE_DIR}/../components/pal/pal_gpio/include  # 被测模块头文件
)
```

**关键**：`mocks` 路径排在所有路径**最前面**（在 CMake 中先列出的优先），确保 `#include "driver/gpio.h"` 优先找到 Mock 版本而非真实 ESP-IDF 头文件。

## 4. 添加新模块的步骤

以添加 `pal_i2c` 测试为例：

### Step 1：创建 Mock 头文件

在 `tests/mocks/driver/i2c_master.h` 中定义 I2C 相关的类型和 FFF Fake 函数。

### Step 2：注册到 fff_globals.c

```c
// fff_globals.c
#define FFF_MOCK_DEFINITIONS
#include "fff.h"
DEFINE_FFF_GLOBALS;

#include "driver/gpio.h"
#include "driver/i2c_master.h"   // 新增
```

### Step 3：编写测试文件

创建 `tests/test_pal_i2c.c`，遵循 Arrange → Act → Assert 模式。

### Step 4：更新 CMakeLists.txt

```cmake
add_executable(pal_test
    unity/unity.c
    fff_globals.c
    test_pal_gpio.c
    test_pal_i2c.c                                    # 新增
    ../components/pal/pal_gpio/pal_gpio.c
    ../components/pal/pal_i2c/pal_i2c.c               # 新增
)

target_include_directories(pal_test PRIVATE
    ${CMAKE_SOURCE_DIR}/fff
    ${CMAKE_SOURCE_DIR}/unity
    ${CMAKE_SOURCE_DIR}/mocks
    ${CMAKE_SOURCE_DIR}/../components/pal/pal_gpio/include
    ${CMAKE_SOURCE_DIR}/../components/pal/pal_i2c/include  # 新增
)
```

## 5. Mock 头文件编写模板

```c
/**
 * @file    driver/i2c_master.h
 * @brief   Mock I2C Master — 宿主机测试用
 *
 * 条件编译：定义 FFF_MOCK_DEFINITIONS 时生成函数体 + 全局变量，
 * 否则仅产生 extern 声明。
 */
#ifndef MOCK_DRIVER_I2C_MASTER_H
#define MOCK_DRIVER_I2C_MASTER_H

#include <stdint.h>
#include "fff.h"

#ifdef __cplusplus
extern "C" {
#endif

/* ---- 1. 类型定义（简化自真实 ESP-IDF 头文件） ---- */
typedef void *i2c_master_bus_handle_t;
typedef void *i2c_master_dev_handle_t;

typedef enum {
    I2C_ADDR_BIT_LEN_7  = 0,
    I2C_ADDR_BIT_LEN_10 = 1,
} i2c_addr_bit_len_t;

typedef struct {
    int      i2c_port;
    int      sda_io_num;
    int      scl_io_num;
} i2c_master_bus_config_t;

/* ---- 2. FFF Fake 函数 ---- */
#ifdef FFF_MOCK_DEFINITIONS
FAKE_VALUE_FUNC1(int, i2c_new_master_bus,
                  const i2c_master_bus_config_t *, i2c_master_bus_handle_t *);
FAKE_VALUE_FUNC3(int, i2c_master_transmit,
                  i2c_master_dev_handle_t, const uint8_t *, size_t, int);
FAKE_VALUE_FUNC3(int, i2c_master_receive,
                  i2c_master_dev_handle_t, uint8_t *, size_t, int);
FAKE_VALUE_FUNC5(int, i2c_master_transmit_receive,
                  i2c_master_dev_handle_t, const uint8_t *, size_t,
                  uint8_t *, size_t, int);
FAKE_VALUE_FUNC1(int, i2c_master_probe,
                  i2c_master_bus_handle_t, uint16_t, int);
#else
DECLARE_FAKE_VALUE_FUNC1(int, i2c_new_master_bus,
                          const i2c_master_bus_config_t *, i2c_master_bus_handle_t *);
DECLARE_FAKE_VALUE_FUNC3(int, i2c_master_transmit,
                          i2c_master_dev_handle_t, const uint8_t *, size_t, int);
DECLARE_FAKE_VALUE_FUNC3(int, i2c_master_receive,
                          i2c_master_dev_handle_t, uint8_t *, size_t, int);
DECLARE_FAKE_VALUE_FUNC5(int, i2c_master_transmit_receive,
                          i2c_master_dev_handle_t, const uint8_t *, size_t,
                          uint8_t *, size_t, int);
DECLARE_FAKE_VALUE_FUNC1(int, i2c_master_probe,
                          i2c_master_bus_handle_t, uint16_t, int);
#endif

#ifdef __cplusplus
}
#endif
#endif /* MOCK_DRIVER_I2C_MASTER_H */
```

**注意事项**：
- FFF 宏的编号 `FAKE_VALUE_FUNC{N}` 中 N = 函数参数个数（不含返回值）
- `FAKE_VALUE_FUNC` 用于有返回值函数，`FAKE_VOID_FUNC` 用于 void 函数
- 类型定义只需写 PAL 层**实际使用到的**字段和枚举值，无需完整镜像
