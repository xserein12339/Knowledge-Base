# 01 — 概念与原理

## 1. 什么是 Mock

在软件测试中，Mock 是指用一个**行为可编程的假函数**替换真实的外部依赖。在嵌入式 C 开发中，外部依赖通常是：

- 外设驱动函数：`gpio_config()`、`i2c_master_transmit()`、`uart_write_bytes()`
- RTOS API：`xQueueSend()`、`vTaskDelay()`
- 硬件抽象层：`HAL_GPIO_WritePin()`、`esp_timer_start_once()`

Mock 函数不操作真实寄存器，而是：
1. 记录**被调用了多少次**
2. 记录**传入的参数是什么**
3. 返回**预设的值**（成功/失败/任意值）

## 2. Mock vs Stub vs Fake vs Spy

这些都是"测试替身"（Test Double），区别在于复杂度：

| 类型 | 行为 | 典型场景 |
|------|------|---------|
| **Stub** | 只返回固定值，无状态 | "不管输入什么，永远返回 ESP_OK" |
| **Fake** | 有简化逻辑的轻量实现 | 用数组模拟 Flash 存储 |
| **Mock** | 可编程行为 + 记录调用信息 | "前 2 次返回 ESP_OK，第 3 次返回 ESP_ERR_TIMEOUT" |
| **Spy** | 真函数 + 调用记录 | 包裹真实函数，记录参数供断言 |

FFF（Fake Function Framework）名字里是 "Fake"，但实际上提供的是 **Mock** 能力：可编程返回值 + 调用计数 + 参数记录。

## 3. 嵌入式 C 的 Mock 核心原理

### 3.1 链接替换

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  test_pal_gpio.c │────▶│   pal_gpio.c     │────▶│ mock gpio_config │
│  (测试用例)       │     │   (被测代码)      │     │ (假的 gpio 函数)  │
└─────────────────┘     └──────────────────┘     └─────────────────┘
                                                         │
                                                  ┌──────┴──────┐
                                                  │ FFF 全局变量  │
                                                  │ .call_count  │
                                                  │ .return_val  │
                                                  │ .arg0_val    │
                                                  └─────────────┘
```

**关键**：编译时不链接 ESP-IDF 库，链接 Mock 实现。`pal_gpio.c` 中的 `#include "driver/gpio.h"` 通过 `-I` 路径优先级找到 Mock 版本。

### 3.2 FFF 的代码生成

FFF 通过宏展开生成：

```c
// ---- FAKE_VALUE_FUNC1 展开后的产物 ----

// 1. 一个结构体，承载调用记录
typedef struct {
    unsigned int call_count;       // 被调用次数
    int          return_val;       // 固定返回值
    int          return_val_seq[8];// 序列返回值
    unsigned int return_val_seq_len;
    const gpio_config_t *arg0_val; // 第 1 个参数快照
    const gpio_config_t *arg0_history[50]; // 历史参数
} fake_gpio_config_t;

fake_gpio_config_t gpio_config_fake;

// 2. 假的 gpio_config 函数体
int gpio_config(const gpio_config_t *arg0) {
    gpio_config_fake.call_count++;
    gpio_config_fake.arg0_val = arg0;  // 记录参数
    if (gpio_config_fake.return_val_seq_len > 0) {
        // 按序列返回
        int idx = gpio_config_fake.call_count - 1;
        if (idx >= gpio_config_fake.return_val_seq_len)
            idx = gpio_config_fake.return_val_seq_len - 1;
        return gpio_config_fake.return_val_seq[idx];
    }
    return gpio_config_fake.return_val;
}
```

### 3.3 条件编译（声明 vs 定义）

Mock 头文件中使用条件宏：

```c
// driver/gpio.h (Mock 版本)

#ifdef FFF_MOCK_DEFINITIONS
  // 仅 fff_globals.c 编译 → 生成函数体 + 全局变量
  FAKE_VALUE_FUNC1(int, gpio_config, const gpio_config_t *);
#else
  // 其他 .c 文件 → 仅 extern 声明，靠链接器找到 fff_globals.c 中的定义
  DECLARE_FAKE_VALUE_FUNC1(int, gpio_config, const gpio_config_t *);
#endif
```

`fff_globals.c`（整个工程有且仅有一个）：

```c
#define FFF_MOCK_DEFINITIONS
#include "fff.h"
DEFINE_FFF_GLOBALS;
#include "driver/gpio.h"  // 此时 FAKE_VALUE_FUNC1 生效
```

## 4. 典型的测试结构

```c
// 1. setUp — 每个测试前重置
void setUp(void) {
    RESET_FAKE(gpio_config);     // call_count=0, return_val=0, arg0_val=NULL
    RESET_FAKE(gpio_set_level);
}

// 2. tearDown — 每个测试后清理
void tearDown(void) {
    pal_gpio_uninstall_isr_service(); // 重置被测代码的 static 状态
}

// 3. 测试 — 4 步模式
void test_set_direction_input(void) {
    // Arrange（配置 Mock 行为）
    gpio_config_fake.return_val = 0;   // 模拟 ESP_OK

    // Act（调用被测函数）
    int ret = pal_gpio_set_direction(4, PAL_GPIO_DIR_INPUT);

    // Assert（验证结果）
    TEST_ASSERT_EQUAL(0, ret);
    TEST_ASSERT_EQUAL_UINT(1, gpio_config_fake.call_count);
}

// 4. 运行入口
int main(void) {
    UNITY_BEGIN();
    RUN_TEST(test_set_direction_input);
    // ...更多测试...
    return UNITY_END();
}
```

## 5. 能测什么，不能测什么

| 能测 ✅ | 不能测 ❌ |
|---------|----------|
| API 返回值的正确传递 | I2C 实际波形 |
| 参数转换是否正确（PAL 枚举 → ESP 枚举） | GPIO 引脚实际电平 |
| 错误路径（底层返回 ESP_FAIL 时 PAL 是否透传） | 中断响应延迟 |
| 幂等操作（重复调用是否被正确拦截） | DMA 传输完整性 |
| 中断回调是否正确注册 | 功耗 |

Mock 测试验证的是**软件逻辑**，硬件时序和电气特性需要 HIL（硬件在环）测试补充。
