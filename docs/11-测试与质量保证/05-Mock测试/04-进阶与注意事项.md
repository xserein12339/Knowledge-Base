# 04 — 进阶与注意事项

## 1. 常见陷阱

### 1.1 指针参数悬垂（Dangling Pointer）

**问题**：FFF 的 `argN_val` 对指针参数只保存地址，不拷贝数据。

```c
int pal_gpio_set_direction(int pin, pal_gpio_dir_t dir)
{
    gpio_config_t cfg = { .mode = GPIO_MODE_INPUT };  // 栈上局部变量
    return gpio_config(&cfg);  // Mock 保存了 cfg 的地址
}  // ← 函数返回，cfg 的内存被释放

// 测试中：
gpio_config_fake.arg0_val->mode  // ← 悬垂指针！未定义行为
```

**解法**：
- **不验证指针参数的内容**（推荐 — 简单安全）
- 使用 `return_val` 和 `call_count` 做间接验证（已经足够有说服力）
- 若确实需要：使用 `custom_fake` 在 mock 函数内拷贝参数到全局缓冲区

```c
// custom_fake 内拷贝参数
static gpio_config_t g_last_cfg;
int my_gpio_config(const gpio_config_t *cfg) {
    memcpy(&g_last_cfg, cfg, sizeof(g_last_cfg));
    gpio_config_fake.call_count++;
    return gpio_config_fake.return_val;
}
// 测试中检查 g_last_cfg.mode
```

### 1.2 静态状态泄漏

**问题**：被测代码中有 `static` 变量，上一个测试的状态影响下一个。

```c
// pal_gpio.c
static bool g_isr_service_installed = false;

int pal_gpio_install_isr_service(int flags) {
    if (g_isr_service_installed) return ESP_OK;  // ← 静态状态
    g_isr_service_installed = true;
    return gpio_install_isr_service(flags);
}
```

**解法**：在 `tearDown` 中显式重置。

```c
void tearDown(void) {
    pal_gpio_uninstall_isr_service();  // 内部将 static 重置为 false
}
```

若被测代码没有提供重置 API，考虑：
- 添加 `_reset_for_test()` 函数（仅测试构建时编译）
- 或使用 `__attribute__((weak))` 从外部覆盖 static 变量

### 1.3 call_count 的类型陷阱

**问题**：FFF 的 `call_count` 是 `unsigned int`，Unity 的 `TEST_ASSERT_EQUAL` 类型推导可能不匹配。

```c
// ❌ 可能失败（类型推导异常）
TEST_ASSERT_EQUAL(1, gpio_config_fake.call_count);

// ✅ 明确指定无符号整数比较
TEST_ASSERT_EQUAL_UINT(1, gpio_config_fake.call_count);
```

### 1.4 忘记在 setUp 中 RESET_FAKE

**问题**：前一个测试的状态影响后一个测试的执行结果。

```c
void setUp(void) {
    // ❌ 漏掉了 RESET_FAKE(gpio_config)
    RESET_FAKE(gpio_set_level);
}

void test_a(void) {
    gpio_config_fake.return_val = -1;
    // ...call_count 变成 1
}
void test_b(void) {
    // ❌ call_count 不是 0！测试 b 可能意外通过或失败
}
```

**解法**：在 `setUp` 中列出**全部**用到的 Mock，形成模板后复制粘贴。

## 2. 持续集成

### 2.1 GitHub Actions 示例

```yaml
name: Unit Tests
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build & Run Tests
        run: |
          cd tests
          cmake -B build -DCMAKE_BUILD_TYPE=Debug
          cmake --build build
          ./build/pal_test -v
```

### 2.2 pre-commit 钩子

```bash
#!/bin/sh
# .git/hooks/pre-commit
cd tests
cmake -B build -DCMAKE_BUILD_TYPE=Debug > /dev/null
cmake --build build > /dev/null 2>&1
if ./build/pal_test > /dev/null 2>&1; then
    echo "Tests passed"
else
    echo "Tests FAILED! Commit rejected."
    exit 1
fi
```

## 3. 扩展 Mock 模块清单

随着项目发展，逐步添加 Mock 头文件：

| 已 Mock | 待 Mock | 依赖链复杂度 |
|---------|---------|-------------|
| `esp_err.h` | — | 0（无依赖） |
| `esp_log.h` | — | 0 |
| `driver/gpio.h` | — | 1（仅 esp_err） |
| — | `driver/i2c_master.h` | 2（esp_err + i2c_types） |
| — | `driver/spi_master.h` | 2（esp_err + spi_common） |
| — | `driver/uart.h` | 2 |
| — | `driver/ledc.h` | 3（esp_err + ledc_types + gpio） |
| — | `esp_cam_ctlr.h` + `esp_cam_ctlr_csi.h` | 5+（多层类型依赖） |
| — | `esp_lcd_mipi_dsi.h` + `esp_lcd_panel_ops.h` | 5+ |
| — | `freertos/FreeRTOS.h` + `freertos/semphr.h` | 4+ |

**建议顺序**：先 Mock 简单外设驱动（gpio → i2c → spi → uart → ledc），最后 Mock 复杂子系统（MIPI CSI/DSI、FreeRTOS）。

## 4. Mock 与真实硬件测试的分工

```
                    Mock 测试              硬件在环 (HIL)
                    ─────────              ─────────────
运行速度            毫秒级                  秒级 / 分钟级
并行能力            多模块并发              串行（一块板子）
覆盖重点            逻辑正确性              时序正确性
错误注入            极易（返回任意错误码）     困难（需物理故障注入）
回归频率            每次 git push            nightly / 发版前
依赖                PC                      目标板 + 调试器 + 外设
```

**黄金法则**：Mock 测逻辑，HIL 测物理。两者互补，不是替代。

## 5. 测试覆盖目标建议

| 测试类型 | 覆盖目标 | 典型用例 |
|---------|---------|---------|
| 正向路径 | 100% | "配置引脚为输出，返回 OK" |
| 错误路径 | ≥ 1 per 错误码 | "底层返回 ESP_FAIL，PAL 透传" |
| 边界条件 | ≥ 1 per 边界 | "占空比 0% / 100%" |
| 幂等操作 | 每种幂等逻辑 1 个 | "重复安装 ISR" |
| 参数转换 | 每种枚举 1 个 | "PAL 上拉 → ESP 上拉枚举" |

> Mock 测试的目标**不是** 100% 代码覆盖率，而是覆盖**所有可独立于硬件验证的逻辑决策点**。
